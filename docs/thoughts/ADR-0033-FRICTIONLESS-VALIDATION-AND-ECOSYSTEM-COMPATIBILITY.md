# ADR-0033: Frictionless Validation And Ecosystem Compatibility

## Status

Accepted for ecosystem planning.

## Context

Ananke is part of a broader runtime ecosystem rather than a standalone demo project.

Current ecosystem repositories:

- [Project Ananke](https://github.com/hourwise/Project-Ananke): execution governance, approval binding, auditability, and recoverable outcomes.
- [Project Mnemosyne](https://github.com/hourwise/Project-Mnemosyne): sister project intended to run alongside Ananke in the same ecosystem.
- [Project Runtime Contracts](https://github.com/hourwise/project-runtime-contracts): shared protocol, types, schemas, and interfaces used by Ananke, Mnemosyne, and future consumers.

Runtime Contracts is intentionally contracts-only. It should contain no runtime logic, engines, databases, persistence, policies, or runtime behavior. Its job is to provide a small, stable, runtime-independent public API that independent runtimes can import to remain protocol compatible.

Current and potential consumers include:

- Ananke.
- Mnemosyne.
- Moira Code.
- Third-party runtimes.

Runtime Contracts should follow these principles:

- Runtime independent.
- No business logic.
- Stable public API.
- Semantic versioning.
- Separate protocol compatibility tracking through `ProtocolVersion`.
- Backwards compatibility where practical.
- Small surface area.

The compatibility purpose is deterministic negotiation. If a future coordinator or client discovers that Ananke speaks Protocol 2 while Mnemosyne only speaks Protocol 1, incompatibility should be known before execution starts.

The following do not belong in Runtime Contracts:

- `ApprovalEngine`.
- `MemoryEngine`.
- `WorkspaceGuard`.
- SQLite or database adapters.
- Audit database implementations.
- Retrieval, policies, or reliability scoring.
- Context packs or memory stores.

The conversation captured in `docs/Chatgpt thoughts.txt` and `docs/more thoughts.txt` raised two architectural concerns:

- Adoption depends on a low-friction first-run experience, not just architecture documents.
- Ananke, Mnemosyne, and shared runtime contracts need compatibility validation, not only isolated package tests.

The discussion also distinguishes gateways from governed runtimes:

- Gateways handle discovery, routing, identity, auth, quotas, and traffic control.
- Ananke handles governed execution after a request reaches a tool boundary.
- Mnemosyne is expected to provide a complementary runtime capability rather than replace Ananke.
- A future coordinator may become useful, but it should not be started until real ecosystem coordination needs appear.

## Decision

Every ecosystem repository should converge on the same first-run and validation shape:

1. Install.
2. Run demo.
3. Run validation.
4. Generate a portable report.
5. Optionally submit an anonymised report.
6. Explore architecture.

Validation must be a product feature, not an afterthought.

Ananke will define and evolve a validation approach that can later be shared with Mnemosyne and Runtime Contracts:

- Bundled test suites for normal, edge, failure, malicious, regression, cross-platform, approval, audit, concurrency, and ecosystem compatibility cases.
- A simple validation entry point for quick, standard, full, and hostile test levels.
- Local reports generated by default.
- JSON and CSV report export.
- User-approved GitHub issue/discussion submission.
- Anonymisation before external reporting.
- Searchable per-test line items, including passing tests.

Runtime Contracts should become the shared source for stable compatibility shapes, including:

- Runtime identity.
- Protocol version.
- Runtime names.
- Capability manifests.
- Validation report schemas.
- Health/readiness contracts.
- Cross-runtime audit/event field contracts where appropriate.

Every validation report should capture at least:

- Project or combined project set.
- Version and commit SHA.
- Test suite version.
- Operating system, OS build, CPU architecture.
- Node, npm, SQLite, and relevant runtime versions.
- Harness/editor/client context where known, such as VS Code, Cursor, Claude Code, Codex, or unknown.
- Model context where known, such as Claude, GPT, DeepSeek, local model, or unknown.
- MCP client/server context where relevant.
- Start and finish timestamps.
- Total, passed, failed, and skipped counts.
- Per-test ID, suite, category, status, duration, failure reason, log pointer, and reproduction command.

Combined ecosystem tests should eventually cover:

- Ananke only.
- Mnemosyne only.
- Runtime Contracts only.
- Ananke plus Mnemosyne.
- Ananke plus Mnemosyne plus Runtime Contracts.

Compatibility tests should check:

- Port conflicts.
- SQLite lock and database conflicts.
- Shared config conflicts.
- MCP namespace and tool-name collisions.
- Memory access boundaries.
- Audit/event ordering.
- Startup and shutdown ordering.
- Resource usage.
- Concurrent requests.
- Failure isolation.

Two ecosystem safety rules follow:

- Ananke failure must not corrupt Mnemosyne memory.
- Mnemosyne failure must not bypass Ananke authority.

## Consequences

- CI should run unit tests, scenario benchmarks, and real demos where feasible.
- Future validation work should produce machine-readable reports rather than only console logs.
- New security issue means a new permanent regression test.
- Runtime Contracts should become the natural home for shared validation report schemas, capability manifests, and compatibility contracts once they stabilize.
- Ananke must not import runtime behavior from Runtime Contracts; it should import only stable protocol definitions.
- Protocol compatibility must be checked early, before execution or combined runtime startup.
- Ecosystem compatibility tests should be added incrementally after Mnemosyne and Runtime Contracts expose stable runnable surfaces.
- A future orchestrator or coordinator remains a possible later project, but is explicitly not a Phase 1 Ananke deliverable.

## Non-Goals

- This ADR does not define Mnemosyne internals.
- This ADR does not introduce automatic telemetry.
- This ADR does not require users to submit reports.
- This ADR does not start a gateway or orchestrator project.
- This ADR does not replace Ananke's existing Phase 1 execution-governance architecture.
