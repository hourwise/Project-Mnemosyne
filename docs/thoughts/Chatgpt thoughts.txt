Yes — and this should be part of ADR-0028.

I would define it as a test matrix, not just “unit tests”.

For every Ananke function/capability, we should have bundled tests in layers:

1. Normal operation
2. Edge cases
3. Failure cases
4. Malicious use
5. Regression tests
6. Cross-platform tests
7. Human approval flow tests
8. Audit integrity tests

Example:

Capability: File write

Normal:
✓ approved write succeeds
✓ audit record created

Edge:
✓ empty file write
✓ large payload write
✓ unicode filename
✓ nested directory path

Failure:
✓ target directory missing
✓ permission denied
✓ disk unavailable

Malicious:
✓ path traversal blocked
✓ symlink escape blocked
✓ payload changed after approval blocked
✓ hidden delete disguised as write blocked
✓ prompt-injected tool description ignored

For Ananke, I’d group them like this:

Policy Engine
Authority Engine
Approval Binding
Tool Router
MCP Adapter
Audit Engine
Outcome Engine
Filesystem Demo
HTTP API
Dashboard
CLI
Report Submission

Each group gets:

Run quick tests
Run standard tests
Run hostile tests
Run all tests

And the dashboard should show:

Approval Binding   31/31 passed
Audit Engine       28/28 passed
MCP Adapter        37/39 passed

This is where it becomes powerful: you are not just asking the community to “try it”.

You are giving them a validation harness.

Clone → install → run validation → submit report

For adoption, this matters a lot. A developer will trust:

“I ran 400 tests locally and here is the report.”

far more than:

“The README says it is safe.”

Yes — that makes ADR-0028 much stronger.

I’d expand it from “frictionless tests” into:

ADR-0028 — Frictionless Validation, Telemetry Reports & Ecosystem Compatibility Testing

Core decision:

Every validation run must produce a portable, anonymised, downloadable report in JSON and CSV, suitable for GitHub issues, release validation, trend analysis, and compatibility tracking.

The report should capture:

Project: Ananke / Mnemosyne / Combined
Version / commit SHA
OS and build
CPU architecture
Node / Python / SQLite versions
Editor or harness: VS Code, Cursor, Claude Code, Codex, etc.
Model used: Claude, GPT, DeepSeek, local model, unknown
MCP client
Test suite version
Started / finished timestamps
Total tests
Passed / failed / skipped
Per-test result
Failure reason
Logs pointer
Reproduction command

The GitHub report summary should look like:

Validation Summary

Date: 2026-07-09

642 tests run
641 passed
1 failed

Failed:
ANANKE-APPROVAL-017
Approval payload mutation was not rejected

Environment:
Windows 11 build xxxxx
VS Code
Claude Code
Node 22
SQLite 3.x
Ananke commit abc123

Then every test, including passes, should be searchable:

{
  "test_id": "ANANKE-POLICY-0042",
  "suite": "Policy Engine",
  "category": "malicious",
  "status": "passed",
  "environment": {
    "os": "Windows",
    "os_build": "xxxxx",
    "editor": "VS Code",
    "model": "Claude",
    "node": "22.x"
  }
}

That gives you exactly what you want later:

Ananke works reliably on:
✓ VS Code + Claude
✓ Linux + Codex
✓ Windows 11 + Claude Code

Known issues:
⚠ VS Code + DeepSeek fails MCP adapter tests
⚠ Windows build xxxxx fails SQLite lock recovery

This is not just testing. It becomes your compatibility intelligence layer.

And yes, absolutely include combined ecosystem tests:

Ananke only
Mnemosyne only
Ananke + Mnemosyne together
Ananke + Mnemosyne + shared runtime contracts
Ananke + Mnemosyne + Moira Code shell

Combined tests should check:

Port conflicts
SQLite lock conflicts
Shared config conflicts
MCP namespace collisions
Tool name collisions
Memory access boundaries
Audit/event ordering
Startup/shutdown order
Resource usage
Concurrent requests
Failure isolation

Important rule:

Ananke failure must not corrupt Mnemosyne memory.
Mnemosyne failure must not bypass Ananke authority.

That line belongs in the ADR.

I’d also define report levels:

Local report: always generated
GitHub issue report: user-approved
Aggregate public stats: anonymised only
Maintainer dashboard: full searchable compatibility matrix

This becomes a huge trust signal. Instead of claiming “it works cross-platform”, you’ll eventually be able to show:

Tested across 184 community environments, 98.7% pass rate, failures linked to known issues.

That is exactly the kind of thing that helps open-source adoption.
